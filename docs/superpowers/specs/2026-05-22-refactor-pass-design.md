# Refactor Pass + Audit Dashboard — Design

**Status:** approved
**Date:** 2026-05-22
**Branch:** sprint-3-foundation

## Context

Лендинг психолога (Next.js 16, React 19, Tailwind v4). ~38 файлов TS/TSX, 2105 строк. MVP без автотестов — верификация через `lint` + `tsc --noEmit` + ручной обход в браузере.

Запрос: тщательный, но консервативный рефакторинг по трём направлениям + наглядный HTML-дашборд «до/после» с рекомендациями на будущее.

## Goals

1. Поднять качество кода: убрать дубли, неиспользуемое, magic-значения.
2. Подтянуть UI-консистентность: spacing/typography/цвета через общие tokens, согласованные обёртки секций.
3. Улучшить производительность и a11y: LCP-картинка, lazy-loading, semantic HTML, ARIA, фокусы, alt.
4. Сгенерировать `docs/refactor-dashboard.html` со сравнением метрик до/после и приоритизированными рекомендациями.

## Non-Goals

- Реструктуризация директорий, переименование публичных props.
- Создание новых абстракций «на будущее» (CLAUDE.md §2).
- Замена/апгрейд библиотек, изменения `package.json`.
- Правки `SPEC.md`, контента (`content/home.ts`), изображений в `public/`.
- **Визуальные изменения UI.** Любая правка, которая сдвигает пиксели/меняет цвет/шрифт — переезжает в раздел «Рекомендации» дашборда, а не в этот рефакторинг.

## Approach — Audit-Driven

Пять фаз. После каждой фазы — verify-gate (`lint`, `tsc`, `build`).

### Фаза 1 — Baseline Audit

Снять метрики «до» автоматически (где возможно). Сохранить в `docs/refactor-baseline.json`:

| Категория | Метрика | Источник |
|---|---|---|
| Code quality | ESLint warnings/errors | `npx eslint . --format json` |
| Code quality | TS-ошибки | `npx tsc --noEmit` (exit code + парсинг) |
| Code quality | Файлов > 100 строк | `wc -l` + фильтр |
| Code quality | Использований `any`/`@ts-ignore`/`@ts-expect-error` | `grep -rE` |
| Code quality | Magic numbers/строк в TSX (heuristic: числовые литералы > 1 в Tailwind/инлайн-стилях вне токенов) | `grep` по паттернам |
| UI consistency | Уникальных Tailwind-комбинаций в `className` (топ-частота повторов) | `grep` + сортировка |
| UI consistency | Использований hard-coded цветов (`#xxx`, `rgb(`) вне `globals.css` | `grep` |
| Perf | Размер `.next/static` (build) | `du -sb .next/static` после `npm run build` |
| Perf | Использований `<img>` вместо `<Image>` | `grep` |
| A11y | `<button>`/`<a>` без `aria-label` И без text-content (иконочные) | manual scan + grep |
| A11y | `<img>` без `alt` | `grep` |
| A11y | Использований `tabIndex={-1}` без причины | `grep` |
| A11y | Отсутствие `<main>`/landmarks | manual scan |

Всё, что нельзя посчитать одной командой, помечается как `manual_scan: <число>` с короткой запиской в комментарии baseline-файла.

### Фаза 2 — Track A: Code Quality

Приоритет правок (только то, что найдёт baseline):

- Удалить неиспользуемые импорты / переменные (ESLint должен видеть).
- Убрать `console.log`/dead code (если есть).
- Вынести повторяющиеся magic-числа (debounce, breakpoint-px, длительности анимаций) в константы — рядом с местом использования, а не в общий `constants.ts`.
- Дубликат **разрешается** выносить в helper только если 3+ идентичных копий и helper не требует придумывать обобщения. Иначе — оставить как есть.

### Фаза 3 — Track B: UI Consistency

- Проверить, что все секции используют одинаковый шаблон контейнера/spacing. Если есть тонкие расхождения (например, `py-20` vs `py-24` между двумя секциями того же уровня) — выровнять под более частый вариант. **Если значения визуально различны намеренно** (по SPEC.md или плану) — не трогать.
- Проверить hard-coded цвета — заменить на Tailwind-токены, если эквивалент существует. Если нет — оставить и записать в рекомендации.
- Согласовать классы между похожими карточками `CaseCard` / `WorkAreaCard` / `ServiceCard` **только в местах, где расхождение случайное** (например, разные `rounded-*` без причины). Преднамеренные различия (по дизайну) — не трогать.

### Фаза 4 — Track C: Perf + A11y

- Hero-картинка: проверить `priority`, `sizes`, `fetchPriority`.
- Картинки ниже первого экрана: `loading="lazy"` (Next/Image делает по умолчанию, но проверить, нет ли `<img>`).
- Заменить `<img>` на `<Image>` где возможно (если есть случаи).
- Semantic landmarks: `<header>`, `<main>`, `<footer>` — должны быть и быть единственными. Проверить `app/(public)/layout.tsx`.
- Все `<button>` с иконкой (карусель, mobile-menu toggle, dialog close) — должны иметь `aria-label`.
- Все `<img>` — `alt`. Декоративные — `alt=""` (явно пустой).
- Фокус-стили: `:focus-visible` на интерактивных элементах. Если глобально не настроены — оставить в рекомендации.
- Контраст: если что-то очевидно вне WCAG AA — отметить, но в Track C не править (это визуальное изменение).

### Фаза 5 — Final Audit + Dashboard

- Снять метрики «после» тем же скриптом, сохранить в `docs/refactor-after.json`.
- Сгенерировать `docs/refactor-dashboard.html`:
  - Hero-summary: 3-4 ключевые цифры улучшений (например, «-12 ESLint warnings», «-8 KB bundle», «+5 ARIA-лейблов»).
  - Таблица метрик: категория → метрика → до → после → дельта (с цветом).
  - Группировка по трекам (A/B/C).
  - Секция «Рекомендации» — нумерованный список:
    - Каждый пункт: приоритет (P0/P1/P2), что сделать, мотивация, риск/затраты.
    - Источники: всё, что нашли в аудите, но не сделали из-за «консервативно» / визуальных изменений / scope.
- HTML — самодостаточный (inline CSS, без зависимостей, без JS), ASCII-friendly, открывается двойным кликом.

## Deliverables

- Изменения в исходниках (точечные, по результатам аудита).
- `docs/refactor-baseline.json` — снимок «до».
- `docs/refactor-after.json` — снимок «после».
- `docs/refactor-dashboard.html` — итоговый отчёт.
- Один git-коммит: `refactor: code quality + UI consistency + a11y/perf pass with audit dashboard`.

## Verification

После каждого трека и финально:
1. `npm run lint` — без ошибок, число warnings ≤ baseline.
2. `npx tsc --noEmit` — без ошибок.
3. `npm run build` — успешно, размер ≤ baseline (или зафиксировать причину роста).
4. Финально — ручной обход `/` в браузере: hero, services, work-areas, cases, FAQ, contact form, mobile-menu, скролл, фокус по табу.

## Risks & Mitigations

| Риск | Митигация |
|---|---|
| «Без визуальных изменений» не гарантировано без screenshot-тестов | Трогать только эквивалентные классы; в спорных случаях — оставлять и переносить в рекомендации. Финальный ручной браузер-обход. |
| Метрики «дублей» субъективны | Считать только то, что можно автоматизировать; subjective-наблюдения идут в рекомендации текстом, не в числах. |
| Один большой коммит сложно ревьюить | Внутри коммита изменения сгруппированы по трекам в commit message; дашборд служит документацией изменений. |
| Аудит-скрипты могут не запуститься | Все команды в плане — стандартный POSIX/npm-stack; при сбое любой метрики — записать `null` и пометить в дашборде. |
